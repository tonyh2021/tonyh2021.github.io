## About V2Ray

> Project V is a set of tools that can help you build your own dedicated base communication network. The core tool of Project V is called V2Ray, which is primarily responsible for implementing network protocols and functions, and communicates with other Project V tools. V2Ray can run standalone or work together with other tools to provide a streamlined workflow.

For more information and resources, please refer to: [https://v2ray.com/](https://v2ray.com/).


V2Ray is more like a platform of tools, supporting a range of protocols, including Shadowsocks, though given recent stability and usability, using SS is not recommended. This article uses the VMess protocol.

#### Start V2Ray

```
systemctl start v2ray
```

#### Configure the Firewall

Do not skip this step, otherwise everything configured afterward will be useless. CentOS 7.0 uses firewall as the default firewall.

> Open a port

firewall-cmd --zone=public --add-port=80/tcp --permanent

> Parameter meanings:

-- zone #scope
-- add-port=80/tcp #add port, format: port/protocol
-- permanent #permanent, without this parameter the change will not persist after a restart

```
[root@ss-1 ~]# firewall-cmd --zone=public --add-port=11234/tcp --permanent
success
```

The `11234` above comes from `PORT:11234` in the installation script output. You can then run `firewall-cmd --list-ports` to view the configuration.

For CentOS 6.0, use iptables instead.

Commands:

```
vi /etc/sysconfig/iptables

-A INPUT -m state --state NEW -m tcp -p tcp --dport 11234 -j ACCEPT

systemctl restart iptables.service # restart firewall to apply the configuration
```

#### Client Installation

I use a Mac, so I chose [V2RayX](https://github.com/Cenmrev/V2RayX). For other platforms, you can choose and download from [here](https://v2ray.com/awesome/tools.html).

#### Server-Side Configuration

V2Ray configuration uses JSON format, which is relatively complex. If you're a developer, you should find it familiar. The default server-side configuration path is `/etc/v2ray/config.json`.

Edit the configuration with `vi /etc/v2ray/config.json`.

```json
{
  "inbounds": [{
    "port": 11234,
    "protocol": "vmess",
    "settings": {
      "clients": [
        {
          "id": "62b1262c-8c69-4f04-aa33-b45bf1eb8414",
          "level": 1,
          "alterId": 64
        }
      ]
    }
  }],
  "outbounds": [{
    "protocol": "freedom",
    "settings": {}
  },{
    "protocol": "blackhole",
    "settings": {},
    "tag": "blocked"
  }],
  "routing": {
    "rules": [
      {
        "type": "field",
        "ip": ["geoip:private"],
        "outboundTag": "blocked"
      }
    ]
  }
}
```

Necessary explanations:

`"port": 11234` is the port mentioned earlier. Note: this is the server's port.
`"id": "62b1262c-8c69-4f04-aa33-b45bf1eb8414",` is the client ID, which must match the client configuration (covered later).
`"level": 1,` and `"alterId": 64` will also be used in the client configuration; just keep them consistent.

#### Troubleshooting and Optimization

Only basic approaches and methods are provided here.

V2Ray has logs to help with troubleshooting. View the client configuration and logs as follows.

![v2ray-03](/images/posts/20190624-v2ray/03.png)

The server-side logs need to be configured first before they are recorded.

Edit the configuration with `vi /etc/v2ray/config.json`.

Add the configuration below. Make sure not to miss the English-mode commas.

```json
{
  "inbounds": [{
    "port": 11234,
    "protocol": "vmess",
    "settings": {
      "clients": [
        {
          "id": "62b1262c-8c69-4f04-aa33-b45bf1eb8414",
          "level": 1,
          "alterId": 64
        }
      ]
    }
  }],
  "outbounds": [{
    "protocol": "freedom",
    "settings": {}
  },{
    "protocol": "blackhole",
    "settings": {},
    "tag": "blocked"
  }],
  "routing": {
    "rules": [
      {
        "type": "field",
        "ip": ["geoip:private"],
        "outboundTag": "blocked"
      }
    ]
  },
  "log": {
    "loglevel": "info",
    "access": "/var/log/v2ray/access.log",
    "error": "/var/log/v2ray/error.log"
  }
}
```

You can then view the server-side logs with the command `tail -f /var/log/v2ray/error.log`. During my first configuration, I forgot to open the firewall port and deduced this by examining the logs.

In addition, since we're using the TCP protocol, HTTP header obfuscation can be applied. See [here](https://v2ray.com/chapter_02/transport/tcp.html) for details.

The key configuration is:

```json
"streamSettings": {
        "network": "tcp",
        "tcpSettings": {
          "header": {
            "type": "http",
            "response": {
              "version": "1.1",
              "status": "200",
              "reason": "OK",
              "headers": {
                "Content-Type": ["application/octet-stream", "application/x-msdownload", "text/html", "application/x-shockwave-flash"],
                "Transfer-Encoding": ["chunked"],
                "Connection": ["keep-alive"],
                "Pragma": "no-cache"
              }
            }
          }
        }
      }
```

Add this inside `inbounds` in the server configuration, at the same level as `port`. Also add it on the client side, but in `outbounds`.

CentOS 8's default kernel version is 4.18.x. Any kernel version above 4.9 can directly enable BBR, so enabling BBR on CentOS 8 is very simple.

```
echo "net.core.default_qdisc=fq" >> /etc/sysctl.conf
echo "net.ipv4.tcp_congestion_control=bbr" >> /etc/sysctl.conf
sysctl -p
```

Then restart the system.

```
sysctl -n net.ipv4.tcp_congestion_control
lsmod | grep bbr
```

If the output contains BBR, it means BBR has been enabled successfully.
